Spring Security下整合多种登录方式设计

发表于2026-04-06|更新于2026-04-06|学习笔记
|总字数:1.9k|阅读时长:9分钟|浏览量:

在Spring Security项目中，其提供的默认的DaoAuthenticationProvider（账号密码登录）往往不足以满足项目需求。

在比较复杂的项目中往往需要多种登录方式并存来进行验证和授权。

本文将以DaoAuthenticationProvider与SmsAuthenticationProvider共存的例子讲解Spring Security如何整合多种登录方式。

Spring Security验证授权调用链

以DaoAuthenticationProvider为例，其验证授权调用链如下：

不难发现若我们要实现短信验证码登录，需要实现SmsAuthenticationToken、SmsAuthenticationProvider、SmsUserDetailsService、SmsUserDetails。

由SmsAuthenticationProvider调用authenticate(Authentication)方法，并在此方法中调用SmsUserDetailsService的loadUserByUsername(String)方法，返回SmsUserDetails。

接下来我们由下而上实现整个链路。

SmsAuthService

这个类用于实现短信验证码状态管理以及验证码比对。

装配StringRedisTemplate以实现短信验证码保存。

generateVerificationCode()方法用于生成6位随机验证码。

sendVerificationCode(String phoneNumber)方法用于调用云服务商的SMS服务并发送短信验证码，同时将发送的验证码载入Redis实现过期自动删除与存储。

verifyCode(String phoneNumber, String verificationCode)方法用于比对验证码是否存在并且正确。

SmsAuthenticationToken

这个类用于实现通行验证Token，类比UsernamePasswordAuthenticationToken。

构造方法SmsAuthenticationToken(String phoneNumber, String verificationCode)用于构造在登录时的验证Token，授权封装为空，交由后续处理，principal、credentials分别与与phoneNumber、verificationCode绑定。

由于本系统的登录鉴权链路采用jwt方案设计，本项目的构造方法SmsAuthenticationToken(Object principal, Object credentials, Collection<? extends GrantedAuthority> authorities)用于构造用户在已登录系统并获得jwt后供JwtAuthenticationFilter过滤使用的Token，读者可以根据项目实际设计方案自行调整此构造方法。

SmsUserDetails

此类用于构造用户信息以供后续授权使用。

读者需要注意，只有phoneNumber、verificationCode、roles字段必须实现，其他字段为可选项，根据项目设计方案自行调整。

由于本项目采用权限组方案，所以Role只负责整合Permisson，本身不作为权限进行使用，所以在getAuthorities()方法中并没有将role信息也载入至authorities。

SmsUserDetailsService

这个类没有难点，它的职责有且只有将用户信息从数据库中带出。

装配CustomerRepo、EmployeeRepo以查询用户信息。

由于作者的项目SMS登录服务同时对后台和前台暴露，loadUserByUsername(@NonNull String phoneNumber)方法对数据库进行了两次查询，读者可以根据自己的项目实际情况调整，大致思路就是去数据库中查询用户信息并返回。

SmsAuthenticationProvider

这个类用来截获SmsAuthenticationToken并进行验证授权。

装配SmsUserDetailsService以进行数据库用户信息拉取，装配SmsAuthService以进行短信验证码核验。

supports(@NonNull Class<?> authentication)方法负责告诉Spring Security在接收到不同类型的AuthenticationToken时该使用哪个AuthenticationTokenProvider。我的SmsAuthenticationProvider只期望接取SmsAuthenticationToken，所以使用此方法将两者进行绑定。

AuthenticationManager

这个Spring Security配置类用于注入你的SmsAuthenticationProvider。

⚠️值得注意的时，在你决定注入自定义的SmsAuthenticationProvider后，若不进行人为干预，Spring Security默认提供的DaoAuthenticationProvider将会被禁用，若需要保留，则需要手动进行注入。

由于Spring Security中如何使用DaoAuthenticationProvider配合数据库中的账号密码进行登录教程诸多，本文章便不再进行normalUserDetailsService的讲解。

总结

Spring Security中由AuthenticationManager对所有的AuthenticationProvider进行管理，接收到不同的AuthenticationToken时会依次调用每个AuthenticationProvider的supports(@NonNull Class<?> authentication)方法进行匹配，匹配成功后执行这个AuthenticationProvider的authenticate(@NonNull Authentication authentication)方法并传导至与数据库交互的UserDetailsService的loadUserByUsername(@Nonnull String username)方法，拉取数据库结果成功后再由AuthenticationProvider决定是否放行，最后返回包含所有信息的Authentication（注意抹除敏感字段如密码等）。
